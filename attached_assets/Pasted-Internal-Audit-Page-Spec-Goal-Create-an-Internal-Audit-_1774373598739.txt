3) Data model

Add 3 new tables.

audit_runs

Stores each audit execution.

export const auditRuns = pgTable("audit_runs", {
  id: serial("id").primaryKey(),
  auditType: varchar("audit_type", { length: 50 }).notNull(), // full, revenue, attribution, funnel, followup, reliability
  status: varchar("status", { length: 30 }).notNull().default("completed"), // running, completed, failed
  overallVerdict: varchar("overall_verdict", { length: 40 }), // trusted, trusted_with_exceptions, not_yet_trusted
  criticalCount: integer("critical_count").notNull().default(0),
  highCount: integer("high_count").notNull().default(0),
  mediumCount: integer("medium_count").notNull().default(0),
  lowCount: integer("low_count").notNull().default(0),
  checksPassed: integer("checks_passed").notNull().default(0),
  checksFailed: integer("checks_failed").notNull().default(0),
  summary: text("summary"),
  createdBy: varchar("created_by", { length: 120 }).default("system"),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
audit_checks

Stores individual automated check results.

export const auditChecks = pgTable("audit_checks", {
  id: serial("id").primaryKey(),
  auditRunId: integer("audit_run_id").notNull(),
  checkKey: varchar("check_key", { length: 120 }).notNull(),
  checkGroup: varchar("check_group", { length: 60 }).notNull(), // revenue, funnel, attribution, followup, reliability, continuity
  title: varchar("title", { length: 200 }).notNull(),
  severity: varchar("severity", { length: 20 }).notNull().default("medium"),
  status: varchar("status", { length: 20 }).notNull(), // pass, fail, warning
  expectedValue: text("expected_value"),
  actualValue: text("actual_value"),
  detailsJson: json("details_json"),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
audit_issues

Stores mismatches / investigation items.

export const auditIssues = pgTable("audit_issues", {
  id: serial("id").primaryKey(),
  auditRunId: integer("audit_run_id"),
  issueCode: varchar("issue_code", { length: 40 }).notNull(),
  area: varchar("area", { length: 60 }).notNull(),
  severity: varchar("severity", { length: 20 }).notNull(),
  expected: text("expected").notNull(),
  actual: text("actual").notNull(),
  suspectedCause: text("suspected_cause"),
  owner: varchar("owner", { length: 120 }),
  status: varchar("status", { length: 30 }).notNull().default("open"), // open, investigating, fixed, ignored
  notes: text("notes"),
  createdAt: timestamp("created_at").defaultNow().notNull(),
  resolvedAt: timestamp("resolved_at"),
});

Optional later:

6) Automated checks to implement
A. Session continuity checks

File:

server/services/auditChecks/continuity.ts

Checks:

paid orders missing session linkage
bookings missing linked session where expected
accepted offer true but no accepted offer slug
won deals with missing source session
orders attributed AI-assisted but no session exists
Example output object
{
  checkKey: "continuity_paid_orders_missing_session",
  checkGroup: "continuity",
  title: "Paid orders missing session linkage",
  severity: "critical",
  status: "fail",
  expectedValue: "0",
  actualValue: "3",
  detailsJson: { orderIds: [11, 14, 18] }
}
B. Funnel checks

File:

server/services/auditChecks/funnel.ts

Checks:

unique session count valid
intent count ≤ chat count
email captured count ≤ intent count or chat count
booked count excludes cancelled
won/paid count not duplicated
qualified definition consistency
C. Revenue checks

File:

server/services/auditChecks/revenue.ts

Checks:

paid orders sum matches revenue endpoint sum
won values match won revenue totals
combined revenue math correct
test orders excluded
refunded/failed payments excluded
average order value formula correct
D. Attribution checks

File:

server/services/auditChecks/attribution.ts

Checks:

accepted offer true implies accepted offer slug exists
accepted offer source valid enum
AI-influenced orders actually have session + recommendation history
direct orders do not have false AI link
source categories are valid
E. Follow-up checks

File:

server/services/auditChecks/followup.ts

Checks:

silent hot queue matches rule
overdue queue excludes closed/won
followupCount increments after mark-as-sent
lastFollowupSentAt present where count > 0
due date extension logic valid
F. Reliability checks

File:

server/services/auditChecks/reliability.ts

Checks:

health endpoint responds correctly
audit log recent writes present
rate limit config visible
days since last lead warning logic valid
days since last digest warning logic valid
7) Verdict rules

The audit service should compute verdicts like this:

Per-check status
pass
warning
fail
Overall verdict
trusted: no critical, <=2 high, most checks pass
trusted_with_exceptions: some high/medium issues but revenue and attribution remain safe
not_yet_trusted: any critical revenue/attribution/session continuity issue
Example logic
if (criticalCount > 0) verdict = "not_yet_trusted";
else if (highCount > 2 || checksFailed > checksPassed * 0.25) verdict = "trusted_with_exceptions";
else verdict = "trusted";

9) Recommended UI behavior
Auto-refresh

Do not auto-run audits repeatedly.

Instead:

summary refresh every 60s
checks/issues refresh on page open
run audits manually via button
Running state

When audit is running:

disable run buttons
show spinner
show “Running audit…”
Empty state

If no audit run exists:

show “No audit has been run yet”
button: Run Full Audit
Severity visuals
Critical: red solid badge
High: orange
Medium: yellow
Low: slate
10) Auth rules

All audit routes should be:

dashboard-auth protected
admin only if you have roles
never public

Health endpoint can remain public:

/api/health

But the Audit tab itself should only consume protected internal endpoints.

11) Replit-friendly implementation notes
Keep it simple first

Avoid background jobs first.
Run checks synchronously inside request/response for Phase 1 of Audit tab.

Keep queries cheap

Start with:

aggregates
small mismatched row samples
limited detail payloads

Do not dump huge transcripts into audit details.

Good first payload limits

For detailsJson, include:

count
top 10 offending IDs
a short explanation

Not full tables.
