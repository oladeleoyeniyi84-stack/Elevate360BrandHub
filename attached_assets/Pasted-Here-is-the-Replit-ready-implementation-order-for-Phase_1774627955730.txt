2. Migration sequence

After shared/schema.ts is updated, run your normal Drizzle migration flow.

Typical order:

npm run db:generate
npm run db:migrate

If your project uses different scripts, use the ones already defined in package.json.

You should expect a migration that adds:

5 new tables
new columns on orders
new columns on digest_reports

Before migrating production, verify:

existing orders.fulfillment_status is not duplicated if it already exists live
the migration only adds missing fields

Because your deployment report says fulfillment_status is already live, this is the one place to be careful. If your local schema is behind production, reconcile that first.

6. Rollback-safe strategy

If you want the safest rollout, publish in two pushes.

Push 1

Backend only:

schema
migration
storage
automation engines
routes
server boot

Verify endpoints first.

Push 2

Frontend:

types
dashboard components

success page
optional route tracker

That reduces the chance of UI + backend failures being mixed together.
