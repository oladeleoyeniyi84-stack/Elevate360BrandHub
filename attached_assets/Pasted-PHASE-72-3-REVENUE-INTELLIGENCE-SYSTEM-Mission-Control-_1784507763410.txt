==================================================
C. REQUIRED CORRECTIVE WORK
==================================================

Correct the Phase 72.2 funnel integrity issue.

Current raw stage counts can produce:

- conversion percentages above 100%
- negative drop-off percentages
- later-stage counts greater than earlier-stage counts

Implement normalized funnel progression using unique visitor/session journeys.

Requirements:

1. Keep raw event counts available for diagnostics.
2. Calculate funnel conversion from normalized unique journeys.
3. Determine the furthest valid stage reached per visitor/session.
4. Count that journey cumulatively through preceding stages.
5. Cap displayed conversion percentages between 0% and 100%.
6. Cap displayed drop-off percentages between 0% and 100%.
7. Add an out-of-order event count.
8. Add duplicate-event diagnostics.
9. Add automated tests proving:
   - no conversion percentage exceeds 100%
   - no drop-off percentage is negative
   - later-stage normalized counts never exceed earlier-stage counts

Do not delete or rewrite Phase 72.2 raw event history.

==================================================
G. COLLECTION API
==================================================

Create:

POST /api/analytics/revenue

Requirements:

- public only where necessary
- server-generated events preferred for payment and sensitive revenue events
- 60 requests per minute per IP for public client events
- JSON only
- closed event vocabulary
- closed revenue-source vocabulary
- amount validation
- currency validation
- metadata maximum 2 KB
- extra keys stripped
- return {"ok":true}
- do not trust client-submitted payment completion values without server verification
- Stripe-derived revenue must come from verified server-side payment/webhook data

Separate trusted server revenue ingestion from untrusted client engagement events when necessary.

Integrate revenue intelligence additively.

Requirements:

- preserve existing payment fulfillment
- preserve webhook idempotency

- correctly represent refunds
- correctly represent failed payments
- do not count initiated checkout as earned revenue
